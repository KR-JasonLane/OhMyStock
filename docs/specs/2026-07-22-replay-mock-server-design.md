# 리플레이 목 서버 설계 (장외 시간 트레이딩 엔진 검증 하네스)

- 작성일: 2026-07-22 (v1 초안)
- 상태: 패널 리뷰 대기
- 관련: Phase 5 Task 8(장중 라이브 스모크)의 **보완재** — 대체재 아님(§2 비범위)

## 1. 목적

장 마감 후에도 트레이딩 엔진(P5 Task 0~7 산출물)을 **코드 수정 없이** 실행·
검증할 수 있는 로컬 목 서버를 만든다. 핵심 아이디어(사용자 제안): 과거의 실제
분봉 데이터를 시간 오프셋으로 재생해, 엔진 입장에서 "지금이 과거의 그 장중
시각"인 것처럼 보이게 한다.

예: 실시간이 7/23 18:42:32이고 재생 앵커가 7/10 09:00이면, 기동 후 1시간
42분이 지난 시점의 시세 요청에는 7/10 10:42:32 시점의 데이터를 응답한다.

부가 가치(이 설계가 노리는 실질 최대 효익): 6a~6c 패널 리뷰로 쌓은 방어선
(체결 판정 레이스 C1, pending 추적, 이중매매 가드, reconcile 분기 ①~⑦)은
실서버에서 **의도적으로 재현하기 어려운** 결함 상황을 전제한다. 목 서버는
이런 결함(전파 지연, 조회 실패, 부분체결, 취소 거부)을 **시나리오로 주입**해
방어선을 반복 검증할 수 있는 유일한 수단이다.

**왜 in-process fake가 아니라 HTTP 레벨인가(개발자 패널 I1 — 비용/효익):**
§9 시나리오를 성격별로 나누면 — ① 429/8005 같은 단발 프로토콜 응답은
mocked-HTTP 단위 테스트로도 가능하며 실제로 어댑터 테스트가 이미 커버한다
(리플레이의 추가 가치는 회귀 확인 수준). ② 전파 지연·부분체결·잔고 반영
지연·취소 거부는 **여러 TR에 걸친 상태적 시퀀싱**(주문 접수→미체결 조회→
잔고 조회가 시간에 따라 다른 답을 줌)이 필요해 단발 mocking으로는 각 TR을
따로 조작하는 취약한 테스트가 된다. ③ 시간 오프셋 재생(진입 창→감시→청산
전 과정)은 in-process fake OrderPort로도 가능하지만, 그 경우 **어댑터·
레이트리미터·TokenManager·HTTP 재시도 경로가 검증에서 통째로 빠진다** —
G1~G3에서 실측한 계약(파이프 구분자, A-프리픽스, 제로패딩, 8005)의 파싱
코드가 실제로 도는 것이 이 하네스의 차별 가치다. 결론: ②③이 본체이고
①은 부수 확인 — HTTP 레벨 구축 비용(별도 앱)은 ②③이 정당화한다.
반대로 순수 도메인 로직의 단위 검증은 기존 fake(OrderPort) 테스트가 계속
담당한다(대체 아님 — 계층별 역할 분담).

## 2. 비범위 (정직한 한계 — 반드시 유지)

- **체결 현실성 검증 아님.** 분봉에는 1분 내 가격 경로·호가창 잔량·체결강도가
  없다. 호가는 합성(§7)이고 매칭은 단순 룰(§8)이다. 슬리피지·호가 소진·
  동시호가 매칭의 실측 검증은 여전히 실제 모의서버 장중 테스트(Task 8) 몫.
- **매수 예약은 근사 재현**(아키텍트 R3 #5 → 트레이더 R3 #1로 강화): 접수
  시점에 미체결 매수 예약분을 차감한 가용현금으로 검사한다(실서버
  ord_alow_amt 차감 재현 — 중복 진입 버그가 리플레이에서 정상으로 보이는
  것 방지). 예약 단가는 접수 시점 고정(이후 시세 결측이 예약액을 침묵
  0원으로 만들 수 없음). 잔여 한계: 시장가 미체결(결함 시나리오)의
  예약액은 접수 시점 현재가 근사. 만에 하나 음수 예수금이 발생하면
  Account.negative_cash_events
  카운터+경고로 표면화 — 카운터≠0인 재생 결과는 kt00001 검증 근거로
  사용 금지.
- **초 단위 타이밍 파라미터의 캘리브레이션 불가**(트레이더 I5): 매칭이 분봉
  low/high 크로스 기준이라 "그 분(60초) 안의 정확히 몇 초에" 크로스했는지는
  데이터에 없다. `exit_limit_timeout_sec=5s`·전파 지연 1~2s 같은 서브-분
  파라미터는 **상태기계 분기 발생 여부**(타임아웃→폴백이 일어나는가)만 검증
  가능하고, 값 자체의 적정성(5초가 맞는가)은 이 하네스로 검증 불가 — 배속
  1.0에서도 존재하는 한계이며 Task 8 실측 몫.
- **실서버 대체 아님.** 리플레이로 아무리 통과해도 Task 8(장중 실주문
  스모크)은 그대로 수행한다. 리플레이는 "장외 시간에 미리 결함을 털어내는"
  가속 장치다.
- WebSocket(0B/0D 실시간)은 P5가 REST 폴링 전용(결정 #27)이므로 비범위.
- 수집/스코어링/분석 파이프라인의 리플레이 구동은 2차(옵션 — §6 카탈로그/
  일봉 엔드포인트를 구현하면 가능해지지만, 1차 목표는 트레이딩 엔진).

> 제안 원문은 "공식문서 기반"이었으나, 이 프로젝트는 공식문서가 틀린 지점을
> 실측으로 여러 번 확정했다(ka10095 구분자: 문서 세미콜론 → 실측 **파이프**,
> trde_tp: 두 자리 표기 → 실측 **한 자리**, ka10081 base_dt 필수, kt00018
> A-프리픽스 등 — CLAUDE.md §5). 공식문서대로만 만든 목은 실서버와 다른
> 목이 되어 테스트를 오도한다.

응답/요청 형태의 우선순위:
1. **실측 사실(CLAUDE.md §5 + `.superpowers/sdd/` 증거 파일)** — G1~G3에서

2. **공식문서(openapi.kiwoom.com)** — 실측하지 않은 필드/엔드포인트의 보충.
3. 사제 랩핑 문서 — 참고만(계약 근거로 사용 금지).

목이 실측과 다르게 동작하는 것을 발견하면 그것은 목의 버그다(반대가 아님).

## 4. 아키텍처

```
backend/
├─ app/                    # 기존 백엔드(변경 없음 — 목적 §1 "코드 수정 없이")
└─ replay/                 # 신규 — app의 **형제 최상위 패키지**(임포트 격리*)
   ├─ main.py              # create_replay_app() — `uvicorn replay.main:app`
   ├─ clock.py             # 시간 오프셋(§5)
   ├─ minute_store.py      # 분봉 저장소 로더(§6 — ⚠️ data.py 금지: 데이터
   │                       #  디렉토리 data/와 이름 충돌 시 패키지 섀도잉,
   │                       #  아키텍트 R2 Critical 실재현)
   ├─ account.py           # 인메모리 계좌(예수금/보유/미체결)
   ├─ matching.py          # 매칭 엔진(§8)
   ├─ faults.py            # 결함 주입(§9 — FaultPolicy 주입 seam)
   └─ api/                 # 키움 엔드포인트 재현(§7)
```

\* 임포트 격리: replay는 `app/`을 임포트하지 않는다(호가 합성의 틱 계산 등은
자체 복제 — 목이 피검 대상 코드를 공유하면 같은 버그가 양쪽에 있어 테스트가
통과해버리는 자기참조 문제). **확정(아키텍트 패널):** 틱 테이블은 **복제**하되
`app.domain.trading.ticks`와의 **값 대조 테스트**(import가 아닌 값 비교)로
드리프트를 조기 발견한다. 격리는 관습이 아니라 테스트로 강제: replay 소스에
`from app`/`import app` 패턴이 존재하면 실패하는 정적 검사 테스트 1건
(아키텍트 I4 — 같은 venv/PYTHONPATH라 런타임은 못 막는다).

**저장소 접근(아키텍트 I6, R2 실측 갱신):** minute_store는 기동 시 분봉
sqlite를 메모리 적재하고 요청 경로에는 DB I/O가 없다 — async 핸들러의
블로킹으로 결함 주입 타이밍(전파 지연 N초 등)이 왜곡되는 것 방지. ⚠️ R1
실측이 예상(수일)을 초과(13개월 — 전량 적재 시 842MB/9.5s 실측)했으므로
**재생 서버는 symbols/since 필터로 앵커 이후 구간만 적재**한다(로더 필터
구현됨). 전량 적재는 오프라인 분석(커버리지 게이트) 전용. R4 기동은
executor 경유 + compose 메모리 상한 명시(R6). 계좌/미체결/faults 상태는
전부 인메모리.

### 4-1. 프로필 전환(자동 전환 금지 — 안전)

`Settings`에 `kiwoom_base_url_override: str | None = None`(및 필요 시 WS
대응 필드) 추가. 설정되면 어댑터가 mock/real URL 대신 이 값을 쓴다.

- **시간 기반 자동 전환은 하지 않는다**(사용자 제안 중 "장 종료 시 자동으로
  목으로" 부분의 수정): 실주문 엔진이 대상 서버를 스스로 바꾸면 "이 주문이
  어느 환경으로 나갔는가"라는 감사 질문이 흐려진다. 전환은 명시적 env 변경 +
  재기동으로만.
- 리플레이 프로필에서는 base URL이 로컬로 고정되므로 실서버(모의 포함)로
  주문이 나가는 것이 구조적으로 불가능하다.
- **override 조합 검증(보안 패널 Critical #1 — validator 필수, 사후 감사가
  아니라 사전 차단):**
  1. `kiwoom_base_url_override`는 **루프백 allowlist**(`http(s)://127.0.0.1`
     /`localhost`)만 허용 — 임의 호스트는 기동 거부. 오타/오설정으로 실전
     앱키·시크릿이 외부 호스트로 전송되는 자격증명 유출 경로 차단.
  2. `kiwoom_mock=False`(실전) + override 설정 조합은 **기동 자체를 차단** —
     리플레이는 태생적으로 mock 전제. 실전 모드 엔진이 목의 가짜 체결을
     실체결로 오인해 실계좌 포지션을 방치하는 역방향 리스크 봉쇄.
  3. override 유효 기동 시 실효 base URL을 WARNING 로그로 명시(이중 방어).
  4. 구현 시 httpx TLS 검증(verify) 기본값이 override 경로에서도 유지되는지
     확인(전역 비활성화 금지 — 보안 Minor).
- **감사 분리(보안 패널 #4 — 미결 → 확정):** `trade_runs`에
  `run_environment: NOT NULL` 컬럼(`mock`/`real`/`replay`) 신설(마이그레이션).
  main.py 조립부가 프로필에서 유도해 create_run에 전달 — JSON 파싱 없이도
  집계/대시보드가 리플레이 런을 구조적으로 필터한다(가짜 체결이 P&L·리스크
  지표를 오염하는 경로 차단). 리플레이 예행은 별도 `DATABASE_URL`(임시 DB)
  사용을 **권장** 운영 관행으로 병기(컬럼은 최종 방어선).
- **엔진 시계**: TradingService/Monitor는 now·calendar를 주입받는다(P5 설계).
  `Settings.replay_time_anchor: datetime | None = None`(**Settings 소속** —
  아키텍트 I8: main.py는 조립 파일이지 설정 SSOT가 아니다) 설정 시 main.py
  조립부가 오프셋 now를 **트레이딩 서비스에만** 주입(수집/스코어링/분석은
  1차 비범위 — 이 셋+트레이딩이 문서 내 "4서비스"다). `replay_time_anchor`도
  override 미설정 시 기동 거부(앵커만 있고 실서버로 나가는 조합 금지 —
  조합 validator에 포함).
- **시계 tz 계약(아키텍트 Critical — 프로덕션 버그 발견의 산물):**
  `replay_now()`와 엔진 주입 now는 **tz-aware**여야 하며, market_calendar의
  전 함수는 인자 시각을 KST로 정규화한다(`_as_kst` — 이 스펙 리뷰에서
  발견된 P5-T7 실버그의 수정으로 이미 반영: UTC-aware now가 정규화 없이
  진입 창·장마감 판정에 쓰여 9시간 어긋나던 결함, 회귀 테스트 포함).

### 4-2. 배포

docker-compose에 `replay` 서비스 추가(선택 기동 — `--profile replay`).
개발 편의상 호스트 직접 실행(`uvicorn ...`)도 지원.

- **포트 바인딩은 `127.0.0.1:PORT:PORT` 명문 요구**(보안 패널 #3 — 기존
  db/backend 관례와 동일). 기본 `ports:` 표기는 0.0.0.0 바인딩이라 무인증
  결함 주입/가짜 계좌 API가 LAN에 노출된다.
- replay는 **프로덕션 backend 이미지와 분리**된 빌드 대상(별도 진입점)로
  구성 — 프로덕션 앱에 `/_replay` 라우트가 섞일 구조적 여지 제거.
- 회귀 테스트로 고정: "프로덕션 FastAPI 앱 라우트에 `/_replay` 프리픽스가
  존재하지 않는다" 단정 테스트 1건(컨벤션이 아니라 테스트가 격리를 보증).
- **compose 서비스 스텁(아키텍트 I5 — 계획서 승격 전 확정):**
  ```yaml
  replay:                     # --profile replay 로만 기동
    build: {context: ./backend, target: replay}   # 별도 스테이지
    ports: ["127.0.0.1:9095:9095"]
    healthcheck: GET /_replay/status
  ```
  backend 컨테이너에서의 도달 주소는 `http://replay:9095`(컨테이너 네트워크),
  **호스트 직접 실행 시** `http://127.0.0.1:9095`. §4-1 override allowlist는
  이 두 형태(`replay` 서비스명 포함 — 컨테이너 내부 한정)를 허용 목록에 반영.
- 실행 형태 통일(아키텍트 I3): 모듈 경로는 `replay.main`(app 하위 아님) —
  §4 트리와 일치.

## 5. 시간 오프셋(clock.py)

- 설정: `REPLAY_ANCHOR`(재생 기준 KST, 예: `2026-07-10T09:00:00`),
  `REPLAY_SPEED`(기본 1.0 — 배속. 10.0이면 실시간 1분에 재생 10분).
- `replay_now() = anchor + (real_now - server_start) × speed`.
- 모든 엔드포인트는 `replay_now()` 기준으로 데이터를 자른다(미래 누출 금지 —
  재생 시각 이후의 분봉/체결 정보는 절대 응답에 나타나면 안 된다. 이것이
  리플레이의 제1 불변식).
- 배속>1은 폴링 주기와의 상호작용(1초 폴링이 재생 10초 간격이 됨)을 감수하는
  개발 편의 기능. **구조적 강제(트레이더 I6 — 문서만으론 사람 실수를 못
  막는다):** ① 목의 모든 응답 헤더와 `/_replay/status`에 현재 speed 스탬프,
  ② 검증 결론/증거 파일에 speed 기록 필수, ③ **speed≠1.0 런은 Task 8 사전
  근거로 사용 금지**를 §10 게이트로 명문화. ④ §8의 전파 지연·fill 억제 등
  결함 주입의 시간 단위는 **벽시계 기준(배속 무관)** — 실서버 지연을 흉내
  내는 값이므로 배속으로 스케일하면 C1 방어선 타이밍이 왜곡된다.

## 8. 매칭 엔진(단순 룰 — 한계 명시)

- 시장가: 재생 시각 현재가(분봉 close)로 즉시 전량 체결.
- 지정가 매수: `limit ≥ 현재가`면 즉시 체결(체결가=현재가 — 더 유리, 실서버
  관례), 아니면 미체결 등록 후 이후 재생 분봉 `low ≤ limit`이 되는 첫
  시점에 **limit 가격으로** 체결. 매도는 대칭(`high ≥ limit`).
  **재평가 규칙(트레이더 R3 #2):** check 시점에 현재가 기준 마켓터블(buy:
  limit≥현재가 / sell: limit≤현재가)이면 **현재가로** 즉시 체결 — 결함 억제
  해제·스프레드 크로스 재개 시 스탠딩 지정가의 실서버 관례. 과거 구간
  크로스(가격이 limit을 스치고 복귀한 경우)만 limit 가격 체결.
- **미체결 지정가의 크로스 판정은 `replay_now()` 진행과 동기**로만 수행 —
  등록 시점에 미래 분봉을 미리 스캔해 체결 시각을 계산해 두는 구현 금지
  (§5 미래 누출 불변식의 매칭판, 트레이더 Minor).
- 체결 반영: 미체결(ka10075)에서 제거, 잔고(kt00018) 갱신, 수수료/세금 계산.
- **주문 전파 지연 기본 재현**: 접수 직후 N초(기본 1~2s, 설정 — **벽시계
  기준, 배속 무관** §5)간 ka10075에 **나타나지 않음** — C1 방어선(전파 유예+
  연속 2회 확인)이 실제로 필요함을 상시 재현(끄면 방어선 무용성 오판 위험).
- 부분체결: 기본 비활성, 시나리오 주입(§9)으로만.
- **⚠️ 익절 지정가 즉시체결 함정(트레이더 I4):** P5 §6-2-b의 익절 1차 주문은
  "지정가=현재가"라 위 룰상 **항상 즉시 체결**된다 — 타임아웃→시장가 폴백
  경로가 정상 매칭으로는 절대 발동하지 않는다("지정가는 항상 잘 팔린다"는
  착시). 폴백 경로 검증은 §9의 "fill 억제" 시나리오가 유일한 실행 수단이며,
  §10 커버리지 표에 이 경로를 별도 행으로 명시한다.
- **⚠️ 진입 지정가도 완전 대칭(트레이더 R4):** P5 진입 지정가는
  `ask − entry_tick_offset×틱`(기본 offset=0 → ask 그대로)이고 목의 합성
  ask는 현재가+1틱이므로, 매수 마켓터블 판정(`limit ≥ 현재가`)상 **항상
  즉시 체결**된다 — `limit_order_timeout_sec`→취소→시장가 재발주 폴백과
  그 안의 poll_unfilled C1 방어가 정상 매칭으로는 절대 발동하지 않는다.
  §9 "진입 지정가 fill 억제" 시나리오가 유일한 실행 수단이며, §10
  커버리지 표에 익절 케이스와 대칭 행으로 명시한다.

## 9. 결함 주입(faults.py) — 이 목의 핵심 가치

HTTP 헤더/관리용 엔드포인트(`POST /_replay/faults`)로 시나리오 온오프:

| 429 레이트리밋 | 어댑터 재시도 경로 |
| 토큰 8005 무효화 | TokenManager 재발급 경로 |

**fill 억제/지연 사용 규율(트레이더 R5 — R7 체크리스트):** ① 익절/진입
지정가 fill 억제로 폴백 경로(타임아웃→취소→시장가)를 검증할 때 유한
창(seconds)이 엔진 타임아웃(exit_limit_timeout_sec/limit_order_timeout_sec)
보다 짧으면 §8 마켓터블 재평가가 창 만료 순간 **자동 체결**시켜 엔진
폴백 코드가 한 번도 실행되지 않고도 '완료'로 보인다 — 표준 사용법은
`seconds=None`(무기한, reset으로만 해제) 또는 엔진 타임아웃보다 충분히
긴 창. ② `api_fault mode="delay"`도 delay_sec이 어댑터 HTTP 타임아웃보다
짧으면 '느린 정상 응답'일 뿐 타임아웃 경로를 자극하지 못한다 — 어댑터
타임아웃보다 크게. ③ 부분체결은 벽시계 interval_sec당 1청크 래칫(폴링
횟수와 체결 진행 분리 — 엔진이 더 자주 폴링해도 결함이 빨리 해소되지
않는다).

**결함 주입 개입 지점(seam — 개발자 I2):** 전역 뮤터블 플래그를 각 모듈이
직접 조회하는 방식 금지(매칭 로직에 테스트용 if 산재 — 규칙 2 위반). 단일
`FaultPolicy` 객체를 api/matching/account가 **주입**받고, 각 결함은 정책
객체의 메서드(예: `delay_for(api_id)`, `suppress_fill(order_no)`)로 조회 —
활성 시나리오 상태는 정책 객체 안에만 존재한다.

**관리 API(개발자 Minor):** `POST /_replay/faults`(바디: `{scenario: str,
params: dict}` — 복수 동시 활성 허용), `GET /_replay/status`(활성 시나리오·
speed·재생 시각), `POST /_replay/reset`. **reset 범위 계약(개발자 I3):**
faults + account(예수금/보유/미체결 초기화) + matching pending 전부. clock
anchor는 **유지**(재생 진행은 리셋과 독립 — 시각까지 되돌리려면 서버 재기동).
관리용 엔드포인트는 `/_replay/*` 네임스페이스로 격리(키움 스펙 오염 금지).

## 10. 검증 계획
