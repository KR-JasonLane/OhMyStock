## 2. 범위 / 비범위

**범위:** 데일리 타임라인 순수 평가기(`domain/orchestration/`),
SchedulerService(asyncio 틱 루프, lifespan 통합), 잡 4종(수집→스코어링→
분석→트레이딩 start) 자동 기동 + 캐치업, `scheduler_events` 감사 테이블
(마이그레이션 0010), API 3종(status/pause/resume), **P5 정정 2건**(§5-1
일일 한도 시딩·진입 래치, §4-c 픽 부재 시 진입 재시도), compose `restart`
정책, 상세 로그(결정 #36).

**비범위 (명시적 이월):** 텔레그램 알림·원격 제어(Phase 8), 대시보드 표시
(Phase 7), 실전 전환 게이트 항목(수수료 재설정·토큰 필수 승격 등 — 회고 §4),
분·초 단위 정밀 크론 표현식(고정 시각 + 거래일 캘린더로 충분), 분산/다중
인스턴스 스케줄링(단일 백엔드 프로세스 전제 — §10-4), 반일장/개장시각
변경일의 캘린더 오버라이드 필드(§10-5 — 운영 절차로 대응, 필요 실측 후 도입).

## 3. 계층 배치

```
domain/orchestration/
  config.py     # ScheduleConfig (시각·잡별 재시도 파라미터 + 기본값)
  timeline.py   # 순수 평가기: evaluate(now, facts) -> tuple[Decision, ...]
  service.py    # SchedulerService (asyncio 틱 루프, 서비스 start() 호출)
store/scheduler_store.py   # scheduler_events insert + 잡별 완료 판정 질의
api/schedule.py            # GET /schedule/status, POST /schedule/{pause,resume}
```

- **timeline.py는 순수 함수**: 입력은 `now`(KST)와 `TimelineFacts`(오늘
  수집/스코어링/분석/트레이딩 몫의 완료·실행·실패 상태 — store가 DB에서
  구성), 출력은 `Decision(job, action, reason)` 목록. I/O·await 없음 —
  창/선행/캐치업/공휴일/자정 경계를 전수 단위 테스트(exit_rules와 동일
  원칙: 판정과 부수효과의 물리적 분리). 4잡 × (창/선행/완료/재시도) 조합이
  한 함수에 몰리지 않도록 **잡별 순수 서브평가기로 분리**한다(개발자 패널
  — 억지 분기 누적 방지).
- SchedulerService는 BackgroundRunService **서브클래스가 아니다** — 그
  베이스는 "1회 실행 run"의 스캐폴딩(at-most-one·run 기록)이고, 스케줄러는
  프로세스 수명과 같이 사는 상주 루프라 모델이 다르다. lifespan에서
  `asyncio.create_task`로 기동, 종료 시 취소. 시계·sleep 주입(테스트 관례).
- 서비스 참조는 구체 클래스 임포트가 아니라 **`start()`/`is_running()`만
  갖는 최소 Protocol로 주입**(개발자 패널 — `_StoreLike` 선례, §9 가짜
  서비스 테스트와 정합).
- 서비스 호출은 **HTTP 우회, `start()` 직접 호출**(결정 #37). 도메인
  conflict_check가 배타를 보장하므로 스케줄러에 별도 배타 로직 없음 —
  `start()`가 `None`(이미 실행/충돌)을 반환하면 다음 틱에 재평가(멱등).
- 시크릿·신규 의존성 없음.

## 4. 데일리 타임라인 (KST, 거래일 캘린더 기반)

거래일 판정은 `market_calendar.is_trading_day`. 비거래일은 전 잡 휴면.
"오늘 몫 완료" 판정은 전부 **DB 기준**(인메모리 아님)이라 재기동/재부팅 후
첫 틱이 곧 캐치업이다(결정 #40 — 별도 캐치업 메커니즘 없음).

| 잡 | 트리거 창 | 선행조건 | 완료 판정 (DB) |
|---|---|---|---|
| 수집 | 거래일 D 19:00 ~ 23:55 | — | D일 시작한 succeeded collection_run 존재 |
| 스코어링 | 수집(D) 완료 직후 ~ 다음 거래일 08:50 | 수집(D) succeeded | reference_date=D인 succeeded score_run 존재 |
| 분석 | 거래일 E 08:20 ~ 09:20 | (연쇄 신선도 게이트가 자체 검증 — 스케줄러는 재검증 안 함) | E일 시작한 succeeded analysis_run 존재 |
| 트레이딩 | 거래일 E 09:00 ~ 15:20 | 엔진 조립됨(TRADE_* 설정) | E일 trade_run이 running이거나, `status='succeeded'`, 또는 **`status='stopped'` AND `stopped_by_kill_switch=true`** |

설계 근거·경계:

- **(a) 수집 19:00**: 당일봉 확정 지연 리스크(P2 운영 노트 — 장 마감 직후는
  미확정 가능). 창 상한 23:55는 자정 넘김 방지(D일 몫 판정 단순화).
- **(b) 스코어링 창이 자정을 넘는 이유**: 수집이 최대 67분(풀 수집)이라 완료
  시각이 가변 — "완료 직후 체인"으로 두되, 실패 재시도가 다음 날 아침까지
  이어질 수 있게 상한을 분석 창 시작 여유 전(08:50)에 둔다. 신선도 게이트가
  낡은 데이터를 자체 거부하므로 시간대와 무관하게 안전.
- **(c) 분석 창 상한 09:20과 진입의 관계**: 진입 창(09:05~09:30,
  TradingConfig)이 끝난 뒤의 분석은 그날 진입에 쓰이지 못하므로 09:20 이후
  캐치업은 스킵하고 사유를 이벤트로 남긴다. 분석이 실패/부재여도 **트레이딩
  start는 막지 않는다**: 기보유 포지션 감시는 픽과 무관하게 필요하다.
  ⚠️ **P5 정정 편입(트레이더 패널)**: 현행 엔진은 `_entries_done` 래치를
  `_enter_positions()` 호출 **전에** 세팅해(`service.py:247`), 09:05 첫
  사이클에 신선한 분석이 아직 없으면 이후 09:10에 분석이 완료돼도 그날
  진입이 영구 스킵된다(조용한 기회 상실 — 분석 창 09:20과 모순).
  **수정**: 신선한 분석 결과 자체가 부재하면 래치를 세팅하지 않고 진입 창
  내 다음 사이클에 재시도한다. 픽이 존재하되 전 후보가 **전략 규칙**(갭
  가드·거래대금 등)으로 탈락한 경우는 정상 판정이므로 현행대로 1회 배치 후
  래치(재평가 없음). 단 전 후보가 **기술적 사유(시세/컨텍스트 부재 — 예:
  일시적 빈 quote 응답)로만 드롭**된 경우는 판정이 이뤄진 게 아니므로
  래치를 세팅하지 않고 재시도한다(트레이더 델타 — degenerate quote 전례,
  CLAUDE.md §5). 이 변경은 TradingService 수정(P5 스펙 §6-3 정정)으로
  Phase 6 태스크에 편입.
- **(d) 트레이딩 완료 판정의 비대칭**: `failed`, 그리고 "장중 비정상 종료"는
  미완료 → 창 내 재기동(reconcile 복구 경로는 실증됨 — P5 회고 §2).
  ⚠️ **`status='stopped'`는 두 경우를 겸한다(개발자 패널 Critical)**:
  킬스위치 정지와 **셧다운 취소**(프로세스 재배포/크래시 — `service.py:211`
  CancelledError 경로도 'stopped' 기록, 단 `stopped_by_kill_switch=false`).
  완료 판정 질의는 반드시 `stopped_by_kill_switch=true`를 함께 검사한다 —
  이를 빼면 장중 크래시 후 재기동이 "완료"로 오판돼 결정 #40이 트레이딩
  잡에서 조용히 무력화된다. 킬스위치 stopped만 완료(같은 날 자동 재기동
  금지 — 운영자 의사 존중). 창 상한 15:20은 마감 직전 무의미한 재기동 방지.
- **(e) 장중 무거운 작업 금지**: 수집 창(19:00~)이 구조적으로 보장.
  스코어링(8초)·분석(65초)은 가볍고, 트레이딩과의 배타는 기존
  conflict_check가 처리(충돌 시 다음 틱 재시도).
- **(f) 트레이딩 start 09:00 ↔ 진입 창 09:05의 5분 버퍼**: reconcile은
  DB read + 소수 포지션 조회 수준이고 주문/시세 TR은 별도 레이트리밋
  버킷(실측 — CLAUDE.md §5)이라 충분(트레이더 패널 확인).

## 5. 스케줄러 동작

- **틱 루프**: `tick_interval`(기본 30초)마다 ① store에서 TimelineFacts
  구성 → ② `timeline.evaluate()` → ③ Decision별 실행(`start()` 호출) +
  이벤트 기록. 시계 오차에 둔감(시각 "도달 후" 판정이지 정확 발화 아님).
- **완료 감지**: 폴링이 아니라 다음 틱의 facts 재구성으로 자연 감지 —
  잡 간 체인(수집→스코어링)도 "선행 완료 판정 충족 시 실행"으로 표현되어
  별도 콜백/대기가 없다.
- **실패 재시도 — 포기는 창 종료뿐(v2 정정)**: `max_attempts` 개념을
  **폐기**한다(패널 합류 지점 — 트레이더 Critical: 트레이딩 잡 3회 포기 시
  전일 보유 포지션의 손절/감시가 그날 전체 마비; 트레이더 Important: 분석
  3회차 08:40이 스코어링 창 상한 08:50을 못 덮는 자기모순). 정책:
  **잡 실행이 failed로 끝나면 잡별 `retry_backoff` 경과 후 창 내 재시도,
  포기는 창 종료 시에만**(`gave_up` 이벤트 + ERROR). 백오프가 시도 횟수의
  자연 상한이다(분석 창 60분/백오프 5분 → 최대 ~12회 — 로그 폭주 없음).
  마지막 실패 시각은 DB(당일 실패 run의 finished_at)로 판정 — 재부팅
  후에도 백오프가 정확하다. 특히 **트레이딩 잡은 백오프 60초**로 짧게 —
  기동 실패가 길어질수록 보유 포지션이 방어선 없이 노출되는 시간이
  늘어나기 때문(감시 공백 최소화가 최우선).
- **pause/resume**: 신규 트리거만 중단(실행 중인 run은 영향 없음 — 정지는
  각 서비스의 stop/킬스위치 사용, `/trade/stop`은 pause와 무관하게 항상
  동작). 인메모리 상태 — 재기동 시 enabled로 복귀(§10-3), 영속 off는
  `SCHEDULER_ENABLED=false`.
- **기동 게이트 연동**: `Settings.run_environment == "replay"`면 스케줄러
  미기동(재생 시계와 실시계 트리거의 혼합 방지 — 리플레이 런은 수동이
  정의상 옳다). `SCHEDULER_ENABLED=false`(env)로도 끌 수 있다(기본 true).
  **테스트 부팅 배제(보안 패널)**: lifespan을 실제로 도는 테스트가 실물
  스케줄러(→ 실 네트워크 수집 트리거 가능)를 띄우지 않도록 **conftest
  autouse fixture로 `SCHEDULER_ENABLED=false`를 기본 주입**하고, 회귀
  테스트로 고정한다(§9). TRADE_* 미설정(엔진 미조립)이면 트레이딩 잡만
  스킵 + `skipped` 이벤트(사유: `engine_not_assembled`), 나머지 잡은 정상.

### 5-1. 같은 날 트레이딩 재기동과 일일 한도 (P5 정정 — app 변경 필요)

`OrderCaps`는 run 단위 인메모리다(`service.py:190` — `_on_accepted`에서 새
인스턴스). 스케줄러가 같은 날 run을 재기동하면 **일일 건수/금액 한도가 0으로
리셋**된다 — §8-1 "일일" 의미론 위반. 대응(개발자 패널 지적 3건 반영):

- **시딩 값**: DB 당일 주문의 **매수·매도 구분 없는** 건수·금액 합산
  (`OrderCaps.check()`는 side 무관 누적 후 SELL만 차단 면제 —
  `service.py:100-116` 실측. "매수만 합산"하면 연속 실행 대비 낮게 복원돼
  한도가 느슨해진다).
- **`buy_blocked` 래치는 시딩 시점에 명시적으로 계산·세팅**한다(시딩 후
  config 임계값과 직접 비교). check()의 부수효과 재트리거에 맡기면 재기동
  직후 진입 게이트가 열린 것으로 읽혀 불필요한 후보 선정을 한 번 거친다.
- **시딩 위치**: `_on_accepted`는 베이스 계약상 동기·무예외·필드 대입
  전용(`background_service.py` 명시) — **빈 인스턴스 생성만 `_on_accepted`,
  DB 시딩은 `_run()` 진입 직후(create_run 다음)** `asyncio.to_thread`로.
- **진입 배치 게이트**: 당일 진입 주문이 이미 있으면(DB 판정) 재기동 run은
  진입 배치를 건너뛴다(이중 진입 방지 1차 게이트 — 재진입 쿨다운·보유 중
  제외는 2차). ⚠️ 알려진 한계(트레이더 Minor): 진입 배치 도중 크래시(3개
  중 1개만 주문)면 나머지 후보는 그날 스킵된다 — 이중 매수 방지를 우선한
  보수적 트레이드오프로 수용, 명시 기록.
- **릴리스 순서(보안 패널)**: 트레이딩 잡 자동 재기동은 이 시딩 구현·회귀
  테스트(§9)가 완료되기 전에 활성화하지 않는다(같은 태스크로 묶거나 선행
  태스크로 — 시딩 없는 재기동은 일일 매수 한도의 실질 무력화).

### 5-2. 설정 일람 (ScheduleConfig 기본값)

| 파라미터 | 기본값 | 파라미터 | 기본값 |
|---|---|---|---|
| collect_at / until | 19:00 / 23:55 | score_until | 08:50 (다음 거래일) |
| analyze_at / until | 08:20 / 09:20 | trade_start_at / until | 09:00 / 15:20 |
| tick_interval_s | 30 | retry_backoff_s (잡별) | collect 600 · score 600 · analyze 300 · trade **60** |

`max_attempts` 없음(§5 — 포기는 창 종료뿐). env 노출은
`SCHEDULER_ENABLED`(기본 true) 하나 — 시각류는 코드 기본값(운영 변경
필요가 실측되면 그때 env 승격, YAGNI. TradingConfig 등 기존 *Config
관례와 일관).

## 6. 저장 스키마 (마이그레이션 0010, insert-only)

| 테이블 | 칼럼 (요지) |
|---|---|
| `scheduler_events` | id, ts(KST), job(collect\|score\|analyze\|trade), action(triggered\|skipped\|retry\|gave_up\|start_rejected), reason(**고정 리터럴 집합**), run_id(nullable — 트리거가 만든 run) |

- **`reason`은 자유 텍스트 금지 — 고정 리터럴 집합(enum)만**(보안 패널
  Important): 예 `window_not_open`, `prerequisite_missing`,
  `engine_not_assembled`, `stale_gate`, `window_closed`, `conflict`,
  `already_running`, `paused`. **예외 원문(`str(exc)`)·심볼·금액은 절대
  넣지 않는다**(ERROR 로그 전용) — `reason`이 무인증 `/schedule/status`로
  노출되기 때문. `/trade/status` warnings의 노출 증가 전례(STATUS 이월
  게이트)와 동일한 스코프 크리프를 계약으로 차단.
- 결정 #36(분석 친화 적재) 적용: "왜 그 시각에 그 잡이 돌았나/안 돌았나"를
  SQL로 복기 가능. `triggered`↔실제 run의 성공/실패는 각 서비스의 run
  테이블에 이미 있으므로 중복 기록하지 않는다(run_id 조인).
- 완료·성공 여부 판정 질의(잡별 "오늘 몫")는 scheduler_store가 소유하되,
  가능하면 **각 소유 스토어에 판정 헬퍼를 노출해 호출**한다(아키텍트 Minor
  — 원시 SQL 4벌 복제 대신; analysis_store가 score_runs를 읽는 read-only
  크로스 조회 선례 준용). 쓰기는 각 서비스 소유 유지.
- 보존 정책: insert-only. 정상 운영 시 일 수십 행이나, **지속 장애 시에는
  잡당 최대 수백 행**(트레이딩 백오프 60초 × 창 6.3시간, retry 이벤트) —
  스토리지 위협 수준은 아니므로 당장 정리 불요, 장기 운영 시 보존기간 도입
  후보로만 기록(아키텍트·보안 Minor. 실패 run 행 누적도 동일 클래스).
- 로그(결정 #36): 모든 Decision을
  `scheduler decision job=<j> action=<a> reason=<r>` 형태로 INFO(트리거)/
  WARNING(스킵·재시도)/ERROR(포기) — grep 한 줄로 하루 타임라인 재구성 가능.

## 7. API

| 엔드포인트 | 인증 | 동작 |
|---|---|---|
| `GET /schedule/status` | 없음 | enabled/paused/dead, 오늘 잡별 상태(완료/대기/포기 + 예정 시각), 최근 이벤트 ≤20 |
| `POST /schedule/pause` | `require_trade_token` | 신규 트리거 중단(자동매매 통제 스위치이므로 trade 스코프) |

무인증 status의 노출은 스케줄 시각·잡 상태·고정 사유 리터럴로 한정
(포지션·금액·심볼 없음 — §6 reason 계약이 이 전제를 지킨다). Phase 7 이월
게이트(무인증 재평가) 목록에 추가.

## 8. 에러 처리

- **잡 실행 실패**: §5 재시도 정책(창 내 백오프, 포기=창 종료). 하류는 연쇄
  신선도 게이트가 자연 차단(스코어링 실패 → 분석이 게이트에서 실패 →
  트레이딩은 픽 부재로 진입 0 — 전 구간 fail-safe가 이미 실증된 경로).
- **start() 거부(None)**: 충돌/이미 실행 — `start_rejected` 이벤트 + 다음
  틱 재평가(백오프 불필요 — 충돌은 일시 상태).
- **틱 루프 자체의 예외**: Decision 1건의 실행 실패가 루프를 죽이지 않는다
  (per-decision try/except + ERROR 로그). 루프 태스크가 죽으면 done
  콜백에서 CRITICAL 로그 + 재기동 — **재기동 예산은 프로세스 수명당 총
  1회**(아키텍트 패널: 매 크래시 1회로 읽히면 30초 간격 무한 크래시-재기동
  루프). 소진 후에는 `/schedule/status`가 `dead`를 영속 표시하며, 회복
  경로는 컨테이너 재시작(§10-2)뿐이다.
- **mock 일봉 지연**(CLAUDE.md §5): 스코어링 신선도 게이트 실패가 아침까지
  반복 → 창 종료 시 `gave_up`(사유 `stale_gate`) — 게이트가 설계대로
  동작하는 것(버그 아님)을 사유 리터럴로 구분 가능하게 남긴다.
- **셧다운 순서(아키텍트 패널)**: lifespan 종료 시 **스케줄러 태스크를 가장
  먼저 취소하고 완전 종료를 `await`로 확인한 뒤** 기존 4-서비스 정리 루프
  → broker.aclose() → engine.dispose() 순서를 밟는다 — 스케줄러가 정리
  중인 서비스에 start()를 재호출하거나 dispose된 엔진에 질의하는 경합
  차단. 실행 중이던 서비스 run은 각자의 CancelledError 경로(P5-T7 실증:
  "성공" 오기록 금지)로 정리.

## 9. 테스트 전략

  09:20, 15:20), 선행 미충족, 완료 시 무동작(멱등), 공휴일/주말 휴면,
  자정 경계(스코어링 D몫의 다음 날 판정), 잡별 백오프(트레이딩 60초),

## 10. 리스크

1. **완전 자동화의 오발 리스크**: 스케줄러 결함이 곧 무인 주문으로 이어질
   수 있다 — 완화: 진입은 기존 방어선 전체(TRADE_* 하드 게이트·갭 가드·
   신선도 가드·킬스위치)를 그대로 통과해야 하며 스케줄러는 어떤 가드도
   우회하지 않는다(start() 호출자일 뿐). 모의 환경 한정 — 실전 전환 게이트
   에서 자동 시작 유지 여부 재확인(회고 §4 게이트 목록에 추가).
2. **단일 프로세스 SPOF**: 백엔드 프로세스가 죽으면 자동화 전체가 멈춘다 —
   완화: compose `restart: unless-stopped`(db/backend, replay 제외) +
   기동 캐치업. ⚠️ **재부팅 시 동작 갭(아키텍트 패널)**:
   `depends_on: service_healthy`는 `compose up` 시에만 평가되고, 재부팅 후
   dockerd의 restart 정책은 의존 그래프 없이 각 컨테이너를 독립 재시작한다
   — backend가 db보다 먼저 뜨면 CMD의 `alembic upgrade head`가 연결 실패로
   종료 → Docker 재시작 백오프(지수, 최대 ~1분)로 재시도 → db 기동 후
   수렴(최악 수 분). **무인 운영 목표상 수 분 지연은 수용**(크래시-재기동
   수렴이 의도된 회복 경로임을 명시). 전제: dockerd가 OS 부팅 시 자동
   시작(systemd enable — 현 서버 확인됨). 호스트 자체 다운은 수용(모의
   단계 — 실전 게이트에서 재평가).
3. **pause 비영속**: 재기동 시 enabled 복귀 — 운영자가 "멈춰둔 줄" 오인
   가능. status에 paused 노출 + 영속 off는 env로 명시(§5). 문서화로 수용.
4. **다중 인스턴스 미고려**: 백엔드 2개를 띄우면 잡이 이중 트리거될 수
   있다(DB 판정과 틱 사이 race). 단일 인스턴스 전제를 명시하고, at-most-one
   은 각 서비스의 인프로세스 가드가 지킨다 — 같은 프로세스 내 이중 트리거는
   구조적으로 불가, 다중 프로세스는 배포 규율로 방지(compose 단일 서비스).
5. **캘린더 정확도 — 두 방향(트레이더 패널 보강)**: ① 정적 공휴일
   테이블(2026)에 없는 임시 휴장 → 트리거돼도 수집은 빈 증분, 트레이딩은
   시세 부재로 진입 불가라 피해 제한적. ② **거래일이지만 개장 시각이
   변경되는 날(수능일 관행적 1시간 지연 개장 등)** — `is_trading_day`도
   `is_market_hours`(09:00~15:30 하드코딩)도 이를 모르므로, 09:05 진입
   창에 미개장 시장으로 진입을 시도하거나 개장 전 스냅샷 시세가 판정에
   섞일 수 있다. 캘린더 오버라이드 필드는 비범위(§2) — **운영 절차로
   대응**: 알려진 시각 변경일(2026 수능일 등)은 **`SCHEDULER_ENABLED=
   false`(영속)를 원칙**으로 우회하고 `/schedule/pause`는 단기/보조
   수단으로만 쓴다(트레이더 델타 — pause는 §10-3 비영속이라 전야 재기동이
   끼면 "가장 필요한 날 아침"에 조용히 풀리는 조합 사고 가능). 연도 캘린더
   갱신 운영 규칙(P5 회고 §4)에 이 확인을 포함.
6. **킬스위치는 당일 스코프(보안 패널)**: 킬스위치 stopped는 "같은 날"
   재기동만 막는다 — **다음 거래일 09:00에는 자동으로 재개**된다. 운영자가
   "멈춰뒀다"고 믿는 자금 경로가 익일 무통지로 재개되는 위험. 완화: 지속
   정지는 `/schedule/pause`(단 §10-3 비영속) 또는 `SCHEDULER_ENABLED=
   false`(영속)를 사용하는 운영 절차를 명시, 실전 전환 게이트에 "전일
   킬스위치 발동 시 익일 자동 재기동 여부 재확인" 항목 추가(§11).
   근본 대응(킬스위치 상태의 DB 영속화)은 Phase 8 원격 제어와 함께 재설계.

## 11. 이후 페이즈 연계

- **Phase 8(텔레그램)**: `scheduler_events` + 각 run 테이블이 알림의 데이터
  원천("오늘 수집 완료/분석 픽 N/진입 M건" 다이제스트, gave_up 즉시 경보 —
  특히 트레이딩 잡 `gave_up`은 보유 포지션 감시 공백을 뜻하므로 최우선
  경보 대상). pause/resume·킬스위치의 원격 트리거 표면도 이 API를 재사용.
- **실전 전환 게이트**: ① 자동 시작(결정 #39) 유지 여부, ② 전일 킬스위치
  발동 시 익일 자동 재기동 정책(§10-6) 재확인 항목 추가.
- **Phase 7(대시보드)**: `/schedule/status`가 타임라인 위젯 원천.
