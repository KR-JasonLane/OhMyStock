### 1-1. 결정 #32에 대한 정정 (v2 — 보안 패널 반영)

**v1에서 "자격증명 분리가 하드 게이트"라고 단정한 서술을 정정한다.** CLAUDE.md §5에
실측된 사실은 **"모의 엔드포인트 + 실전 키 페어 → `[8030]` 거부"** 이 **한 방향뿐**이다.

- 이는 **설정 실수(mismatch)를 잡는 트립와이어**이지, **실전 상태의 런타임 안전장치가
  아니다.** `client.py:25`의 `base = MOCK_BASE if settings.kiwoom_mock else REAL_BASE`가
  보여주듯 `kiwoom_mock` 플래그와 앱키 종류는 코드 레벨에서 독립 설정 가능하다.
  `kiwoom_mock=False` + 올바른 실전 키가 정상 구성되면(=실전 전환이 제대로 완료된
  상태) `[8030]` 게이트는 전혀 개입하지 않는다.
- **모의 키 → 실전 엔드포인트 거부 여부(대칭성)는 미검증**이다 → §4 G4로 추가.

사용자 결정 #32(모의 안정화 후 전환) 자체는 유효하다. 다만 그 결정을 뒷받침하는
논거로 v1이 "하드 게이트"를 과장했다. 실전 전환 이후의 버그를 막는 것은
자격증명이 아니라 §8-1의 버그 봉쇄 장치와 §8-2의 수동 승인이다.

### 1-2. 버그 봉쇄가 P5 범위인 이유 (결정 #32 보충)

감시 루프가 ~1초 주기로 도는 구조에서 조건 판정 버그는 곧바로 주문 폭주가 된다.
모의에서도 ① 레이트리밋(TR당 ~1req/s)을 소진해 **정작 손절 주문이 뒤로 밀리고**,
② 체결 이력이 오염되어 "모의에서 안정화됐다"는 판단 근거 자체가 무너진다.
따라서 §8-1은 실전 전환과 무관하게 P5 범위다.

## 3. 계층 배치

```
core/background_service.py    # 정지 계약 추가 (선행 확장, §6-5)
domain/trading/
  config.py       # TradingConfig — 임계치·한도·타이밍·비용·검증(__post_init__)
  models.py       # Order, Fill, TradePosition, PositionState, EntryPhase, ExitReason
  costs.py        # 매매비용 계산 (시장별 세율·매수매도 비대칭·ETF 비과세)
  selection.py    # 진입 후보 선정 (순수 함수: 필터→슬롯→사이징)
  exit_rules.py   # 청산 판정 (순수 함수: 가격·peak → ExitDecision + new_peak)
  entry.py        # EntryExecutor — 지정가 발주 → 미체결 감시 → 시장가 폴백
  monitor.py      # PositionMonitor — 폴링 루프, 판정 위임, 청산 주문 발행
  reconcile.py    # 기동 시 DB↔브로커 상태 대조 (순수 판정 + 조회는 주입)
  service.py      # TradingService (확장된 BackgroundRunService 서브클래스)
adapters/kiwoom/broker.py     # OrderPort 구현 (주문·미체결·다종목시세)
store/trading_store.py        # 주문/체결/포지션 리포지토리
api/trade.py                  # 4개 라우트 (POST는 require_trade_token)
```

**순수 함수 3개(`costs`·`selection`·`exit_rules`)를 부수효과(`entry`·`monitor`)와
분리하는 이유:** 자금 손실과 직결되는 판정 로직(어떤 종목을 얼마에 사고, 언제
파는가)을 폴링·주문 발행과 얽으면 경계값 전수 테스트가 불가능하다. 판정(순수)과
집행(부수효과)을 물리적으로 가른다.

**`models.py`를 별도 파일로 두는 이유(개발자 Minor):** 트레이딩은 도메인 타입이
많고(주문·체결·포지션 + 3개 상태 enum), 상태기계가 여러 모듈(`entry`/`monitor`/
`reconcile`)에서 공유된다. `scoring`/`analysis`와 달리 타입을 한 실행 모듈에 둘 수
없어 분리한다.

## 5. 포트 확장 (ISP — 개발자 #4)

기존 `BrokerPort`(시세/캔들/잔고/종목 7개 메서드)는 **그대로 둔다.** 주문 계약을
여기 추가하면 시세만 필요한 `CollectionService`/`ScoringService`가 개념적으로 주문
실행 계약에 의존하게 된다(ISP 위반). 별도 포트를 신설한다:

```python
class OrderPort(Protocol):
    async def get_quotes(self, symbols: list[str]) -> list[MarketData]: ...
    async def place_order(self, req: OrderRequest) -> OrderAck: ...
    async def cancel_order(self, order_no: str, symbol: str) -> OrderAck: ...
    # 전량 취소 고정(cncl_qty="0" — G2 실측 계약). 부분 취소는 P5 비범위
    # (부분체결 시 체결분 인정 + 잔량 전량 취소 — §6-1)라 quantity 불필요(T4 구현 정합)
    async def get_open_orders(self) -> list[OpenOrder]: ...
```

`KiwoomBroker` 단일 구현체가 `BrokerPort`와 `OrderPort`를 **둘 다** 만족한다(비용
거의 0). `TradingService`/`EntryExecutor`/`PositionMonitor`는 `OrderPort`에만 의존.

- `MarketData`는 기존 `Quote`를 **합성**한다(개발자 Minor — 필드 중복 방지):
  `MarketData(quote: Quote, bid: int, ask: int)` 형태로, 현재가/등락율/거래량은
  `Quote`를 그대로 감싸고 **최우선 매수(bid)/매도(ask)호가만 추가**한다(트레이더
  #7 — "몇 틱" 지정가 산정에 필요). G1에서 `ka10095`가 호가를 주는지 확인하고,
  안 주면 별도 호가 TR을 `get_quotes`에 합성하거나 진입 지정가 전략을 재검토
- 키움 호가구분 코드(`trde_tp` 단일자리 `"0"` 지정가/`"3"` 시장가 — G2/G3 실측)는
  **어댑터 내부 매핑 테이블에만** 존재. 도메인은
  `OrderStyle.LIMIT`/`MARKET` enum만 안다 (CLAUDE.md §3 — 브로커 교체 가능성)
- `OrderAck`/응답은 **바디만** 도메인·저장소로 전달한다. `Authorization` 헤더·토큰
  문자열은 어느 계층에도 넘기지 않는다 (§9, 보안 C1)

### 6-1. 포지션 상태기계

```
        진입 신호 (§6-3)
           │
   ┌───────▼────────┐  EntryExecutor (§6-3)
   │ PENDING_ENTRY  │   entry_phase ∈ {LIMIT_SUBMITTED, CANCEL_REQUESTED, MARKET_SUBMITTED}
   └───────┬────────┘
           │ 체결분 확정 (부분체결이면 체결 수량만 인정, 잔량 취소)
   ┌───────▼────────┐
   │    ENTERED     │◀── 재기동 reconcile (§6-6) 로 이 상태에 재진입 가능
   └───────┬────────┘
           │ exit_rules (§6-2) 청산 조건 충족  |  또는 max_holding_days 초과(#34)
   ┌───────▼────────┐  청산 주문 (§6-2-b)   ┌──────────┐
   │    EXITING     │─────────────────────▶│  CLOSED  │
   │  exit_phase ∈  │                      └──────────┘
   │ {LIMIT_SUBMITTED, MARKET_SUBMITTED}  (익절 지정가 경로만; 손절·트레일링·기간초과는 시장가 단일)
   └────────────────┘

   실패 경로: PENDING_ENTRY ─(취소·재주문 모두 실패)─▶ ENTRY_FAILED
              EXITING ─(매도 재시도 소진)─▶ EXIT_FAILED  ← 알람, 절대 침묵 금지
```

- **`PENDING_ENTRY`의 서브상태 `EntryPhase`를 명시적 enum으로 둔다**(개발자 #2):
  `LIMIT_SUBMITTED`(지정가 발주됨) → `CANCEL_REQUESTED`(타임아웃, 취소 요청) →
  `MARKET_SUBMITTED`(시장가 재발주). 이 값이 DB에 있어야 재기동 시 복구 지점을
  안다(§6-6). enum 없이 구현하면 `entry.py`에 임시 플래그가 쌓인다
- **`EXITING`의 서브상태 `ExitPhase`도 대칭으로 둔다**(v3, 개발자 #1): 익절
  청산은 §6-2-b에서 "지정가 → 5초 → 시장가"라는 진입과 동일한 2단계 폴백을
  쓰므로 `LIMIT_SUBMITTED` → `MARKET_SUBMITTED` 서브상태가 필요하다. 손절·
  트레일링·기간초과 청산은 시장가 단일이라 서브상태가 없다(`ExitPhase`는 익절
  경로에만 존재). 이 값도 DB에 영속화해 재기동 복구(§6-6)에 쓴다 — 진입만
  정교화하고 청산을 단일 화살표로 두면 이익 확정 경로에 같은 클래스 결함이 남는다
- **부분 체결**은 체결분만 포지션으로 인정하고 잔량은 취소한다. 전량 대기는
  하지 않는다 — 대기 중 이미 보유한 수량이 손절 감시 밖에 놓인다
- **`EXIT_FAILED`는 절대 조용히 넘기지 않는다.** 재시도 소진 시 명시적 실패
  상태로 고정하고 상태 API(§6-7)에 노출한다

### 6-2. 청산 판정 (`exit_rules.py`, 순수 함수)

**시그니처(개발자 #3 — peak 갱신도 순수 계산에 포함):**
```python
def evaluate_exit(entry_price, current_price, peak_price, trailing_active,
                  held_business_days, config)
    -> ExitDecision   # (reason: ExitReason | None, new_peak: int, new_trailing_active: bool)
```
`peak` 갱신(`max(peak, current)`), 트레일링 활성화 판정, 2단계 폭 선택,
보유기간 초과 판정이 모두 판정 함수 안에서 계산돼 반환된다. `monitor.py`는
반환된 값을 저장만 한다 — 트레일링 로직이 순수 테스트 범위 밖으로 새지 않는다.
(`held_business_days`는 `market_calendar` 기반으로 monitor가 계산해 주입 —
캘린더 조회라는 부수효과를 순수 함수 밖에 둔다.)

**판정 우선순위.** 0단계(보유기간 초과 강제 청산)가 최상위이고, 그 아래
가격 기반 판정(1~3번) 중에서는 **손절이 항상 최우선**이다 — 급변동으로 손절선과
트레일링선을 동시에 뚫는 경우 손절을 택한다(실행은 둘 다 시장가라 결과는 같고
`exit_reason` 라벨링만 결정 — 트레이더 #13, 개발자 v3 Minor).

0. **보유기간 초과**(결정 #34, 최우선 강제 청산) — 진입일로부터 영업일 기준
   `max_holding_days` 경과 시 시장가 청산. 무기한 보유의 다중 갭 누적 차단
1. **손절** — `current ≤ entry × (1 - stop_loss_pct)`
2. **트레일링 스톱** — `trailing_active`이고 `current ≤ peak × (1 - 현재_trailing_폭)`
   - 활성화: `peak ≥ entry × (1 + trailing_activate_pct)`를 한 번이라도 충족하면
     `trailing_active=True`로 래치(이후 계속 유지)
   - **가변 폭(결정 #35, 트레이더 ②·v3.1 선형 보간):** `현재_trailing_폭`은
     `peak` 기준으로 순수 함수 안에서 결정된다. 활성화 경계의 노이즈성 되돌림에
     조기 청산되는 휘핑쏘를 막기 위해 **활성화 직후엔 넓은 폭, 안착 후엔 좁은
     폭**을 쓰되, **계단 함수가 아니라 선형 보간**한다:
     - `peak ≤ entry × (1 + trailing_activate_pct)` 이하: 아직 트레일링 미활성
     - `peak ≥ entry × (1 + trailing_widen_until_pct)` 이상: 좁은 폭 `trailing_stop_pct`
     - 그 사이 구간: 두 폭을 **선형 보간** —
       `폭 = wide − (wide − narrow) × (peak_수익률 − activate) / (widen_until − activate)`
       (`wide=trailing_stop_wide_pct`, `narrow=trailing_stop_pct`)
     - 근거(트레이더 v3): 계단 함수는 전환점(예: +8%)에서 보호선이 불연속으로
       점프해(넓은 폭 → 좁은 폭 순간 +2%p대 급등), 신고점 직후 정상 눌림목에
       털리는 새 휘핑쏘를 만든다. 선형 보간은 전환 불연속을 제거하면서 "직후
       여유 → 안착 후 타이트" 취지를 보존한다. 계수 캘리브레이션은 §11-8 이월
3. **고정 익절** — **`trailing_active`가 False일 때만** `current ≥ entry × (1 + take_profit_pct)`
   (결정 #29 v2 — 트레일링 활성화 후 고정 익절 **해제**. 트레이더 #4: 단조 상승장에서
   고정 익절이 트레일링보다 먼저 걸려 추세를 놓치는 구조 제거)

**기준 가격(그로스/넷 경계 — 트레이더·개발자 Minor):** 위 `stop_loss_pct`/
`take_profit_pct`/`trailing_*`는 모두 **비용 미반영 그로스 가격 기준**이다. 매매
비용은 청산 트리거가 아니라 P&L 기록(§7·§9)에만 반영한다. 근거는 §7 참조.

#### 6-2-b. 청산 주문 체결 방식 (트레이더 #1 — Critical)

청산은 진입과 **다른** 체결 전략을 쓴다. 진입의 60초 지정가 대기를 손절에 그대로
적용하면, 급락 중 지정가가 시장을 못 따라가 60초를 허비하고 시장가로 넘어갈 땐
이미 -5%를 크게 초과한다 — 트레일링의 취지와 정면 배치.

| 청산 사유 | 1차 주문 | 폴백 |
|---|---|---|
| **손절 (stop_loss)** | 시장가 즉시 | (해당 없음) — 속도 최우선 |
| **트레일링 스톱** | 시장가 즉시 | (해당 없음) — 이미 고점 대비 하락, 속도 우선 |
| **보유기간 초과(#34)** | 시장가 즉시 | (해당 없음) — 강제 청산 |
| **고정 익절** | 지정가(현재가), `ExitPhase.LIMIT_SUBMITTED` | `exit_limit_timeout_sec`(5초) 미체결 시 시장가, `ExitPhase.MARKET_SUBMITTED` |

근거: 하방 청산(손절·트레일링)과 기간초과는 슬리피지보다 **미청산 리스크**가
크므로 즉시 시장가. 익절은 이미 수익 구간이라 짧은 지정가로 슬리피지를 줄이되,
5초 안에 안 되면 시장가로 확정한다.

**고정 익절 분기의 성격(트레이더 Minor):** 결정 #29 v2로 트레일링 활성화 후엔
고정 익절이 해제되므로, 이 분기는 **정상적 연속 상승 경로에서는 거의 발동하지
않는 백스톱**이다 — `trailing_activate_pct`(5%)를 건너뛸 만큼 폴링 간격 사이에
급등해 곧장 `take_profit_pct`(10%)에 닿는 경우에만 트레일링 미활성 상태로 익절이
걸린다. 의도된 안전장치이며 결함이 아니다.

#### 6-2-c. `TradingConfig` 검증 (보안 #5)

`Settings`의 `model_validator` 선례(config.py:37-49)를 따라 `__post_init__`에서
fail-fast 검증: `0 < stop_loss_pct < 100`, `0 < take_profit_pct`,
`trailing_activate_pct ≥ 0`, `0 < trailing_stop_pct ≤ trailing_stop_wide_pct < 100`,
`0 < max_capital_pct ≤ 100`, `max_positions ≥ 1`, `max_holding_days ≥ 1`,
`poll_interval_sec > 0`, `max_single_order_krw > 0`, 등.
`AnalysisConfig`(검증 없는 순수 dataclass)와 달리 이 값들은 손절가·수량·감시
주기를 직접 결정하므로 검증 부재의 대가가 금전적이다.

**실전 모드 스코프 토큰 강제(v3 보안 #3 — 잔여 구멍 차단):** `Settings`의 기존
실전 모드 검증자(`kiwoom_mock=False`일 때 `api_write_token` 필수, config.py:37-49)를
**확장**해, 실전 모드에서는 `api_trade_token`도 별도로 설정됐고(`is not None`)
`api_write_token`과 **다른 값**임을 강제한다(둘이 같으면 스코프 분리가 명목뿐).
이것이 없으면 결정 #33의 스코프 분리가 "설정하면 되는 옵션"으로만 남아, §1-1이
"실전 이후 안전은 §8-1·§8-2가 담당한다"고 못박은 논지와 어긋난다. 이 강제로
스코프 분리가 실전 전환 시 "설정 안 하면 기동 자체가 안 되는" 하드 게이트가 된다.

### 6-3. 진입 실행 (`selection.py` 순수 선정 → `entry.py` 집행)

**대상:** `GET /analyze/latest`의 최종 리스트(P4 §12).

**`selection.py`(순수 함수 — 개발자 #7): 어떤 종목을 얼마에 살지 결정.**
0. **갭 가드**(트레이더 #8, C에서 승격) — 각 후보의 현재가가 신호 기준가(P3
   as-of 종가) 대비 `signal_gap_guard_pct`(3%) 이상 벌어졌으면 **그 후보 제외**.
   일봉 신호의 전제가 이미 깨진 종목을 사지 않는다
1. **진입 제외 필터** — `Instrument.audit_info`/`state`가 정상이 아닌 종목
   (거래정지·관리종목 등) 제외. 이 필드는 Phase 3부터 저장 중
2. **유동성 필터**(트레이더 #10) — 최근 평균 거래대금이 `min_avg_trading_value_krw`
   (§6-2 설정값) 미만인 종목 제외(저유동성 종목의 시장가 폴백 슬리피지 방지).
   **데이터 소스: 저장된 일봉(close × volume)의 최근 N영업일 평균** — 별도 TR
   호출 없이 수집된 캔들로 계산(Phase 2 자산 재사용)
3. **보유 중복 제외** + `max_positions` 잔여 슬롯 계산
4. **사이징(고정 슬롯 — 트레이더 #5, C5):**
   `슬롯 크기 = min(가용자금 × max_capital_pct, 예수금) / max_positions`
   **분모는 그날 후보 수가 아니라 고정 `max_positions`(=5).** 후보가 5개 미만이면
   나머지는 현금으로 남긴다. 이렇게 해야 종목당 비중이 회차와 무관하게 일정하고
   결정 #30의 분산이 유지된다
5. **수량 = 슬롯 크기 × (1 - 수수료 버퍼) ÷ 현재가**, **내림**(트레이더 #9 —
   수수료 포함 시 주문가능금액 초과로 거부되는 것 방지). `Deposit.available`
   기준. 0주면 스킵

**`entry.py`(부수효과): 발주·미체결 감시·폴백.**
6. 지정가 = 최우선 매도호가 기준 매수 유리 방향 몇 틱 — **호가단위로 반올림**(§4).
   `MarketData`의 호가 필드 사용(§5)
7. `entry_window`(09:05~09:30) 밖에서는 **신규 진입 안 함.** 단 **청산 감시는 장중
   내내 동작**(진입 창이 닫혀도 손절은 멈추지 않는다)
8. 발주 → `limit_order_timeout_sec`(60초) 미체결 → 취소(`CANCEL_REQUESTED`) →
   시장가 재발주(`MARKET_SUBMITTED`)
   - **체결 판정 계약(6a 트레이더 패널 C1):** 주문 부재(ka10075에 없음)는
     '체결'과 '미체결 시스템 미전파'를 구분하지 못한다(접수 TR과 조회 TR은
     다른 백엔드 계층). 오판 방어 3중 — ⓐ 발주 직후 첫 폴 전 1 interval
     전파 유예, ⓑ 한 번도 관측 못 한 주문의 부재는 성공 조회 **연속 2회**
     확인 후에만 체결 판정(관측된 주문의 부재는 즉시 체결 — 우리가 취소하지
     않았으므로), ⓒ 잔여 리스크는 Task 7이 진입 직후 잔고 대사(kt00018)로
     수량·평단 확정(유령 포지션 즉시 해소)
   - **지정가 취소 실패 = 폴백 중단**(이중 매수 가드): 취소 실패는 주문 상태
     불명(마지막 폴 이후 체결 가능) — 시장가 재발주를 강행하면 이중 포지션.
     `requires_reconcile` 실패로 표면화하고 잔고 대조로 위임
   - **시장가 재발주 직전 시세 재조회**(트레이더 I1): 관측 ask는 타임아웃
     동안 낡았고 미체결 자체가 급변 신호 — caps 재검증·평단 추정 모두
     `get_quotes` 재조회 값 사용(실패 시 stale 폴백+경고, 발주는 막지 않음)
   - **시장가 부분체결도 잔여 취소**(§6-1 동일 원칙) — 감시 밖 지연 체결 =
     미추적 수량. 취소 실패는 별도 사유 + `requires_reconcile` 표식(I3)
   - **`EntryOutcome.requires_reconcile`(구조화 필드)**: 체결 여부 불명·취소
     실패 케이스의 표식. Task 7은 이 필드로 **즉시 미니 reconcile**(잔고
     대사)을 트리거한다 — 재기동 시점까지 대기 금지(트레이더 I2). 사유
     문자열 매칭 분기는 금지(개발자 패널)
   - **⚠️ requires_reconcile=True는 확정 ENTRY_FAILED가 아니다**(아키텍트
     #1): ENTRY_FAILED는 §6-6 재기동 스캔 집합(PENDING_ENTRY/ENTERED/EXITING/
     EXIT_FAILED) **밖**이므로, 미니 reconcile 완료 전에 ENTRY_FAILED로
     영속하면 미니 reconcile이 크래시/누락될 경우 실보유 종목이 어떤 재기동
     스윕에도 다시 걸리지 않는다. 마지막 persist된 EntryPhase(CANCEL_REQUESTED
     /MARKET_SUBMITTED)를 유지한 채 미니 reconcile이 최종 상태(ENTERED/
     ENTRY_FAILED)를 결정한다

### 6-4. 감시 루프 (`monitor.py`)

- 보유 종목 전체를 `get_quotes`로 1회 조회 → 각 포지션에 `exit_rules` 적용 →
  청산 결정 시 §6-2-b 방식으로 주문
- **조회 실패는 "가격 변동 없음"이 아니다.** 연속 실패가 `quote_failure_threshold`
  초과 시 경고 상태로 전환하고 상태 API에 노출(경고 중에도 폴링은 계속)
- **보유 중 거래정지 전환 구분**(트레이더 #12) — 특정 종목만 조회가 계속 실패하면
  `list_instruments`의 최신 `state`를 확인해 "거래정지로 인한 예상된 실패"와
  "네트워크 이상"을 구분한다. 거래정지 종목은 자동 청산 불가이므로 별도 경고
- **동시호가·VI 구간**(트레이더 #6·#14) — 15:20~15:30은 단일가 매매라 주문이 즉시
  체결되지 않고 15:30 매칭까지 대기한다. 이 구간과 VI 발생(연속 미체결로 감지)
  시에는 재시도 주기를 늦춰(백오프) 레이트리밋을 낭비하지 않는다. "폴링 후 즉시
  체결" 가정을 이 구간엔 적용하지 않음
- **장 운영시간 밖에서는 폴링 중지.** 정확한 운영시간·휴장일은 확장된
  `market_calendar.py`(정적 공휴일 테이블, §4)를 쓴다
- **일별 종료(아키텍처 #4):** 정규장 마감(15:30) 도달 시 `_run()`이 **정상
  반환**하고 `_running`이 복귀한다. 다음 거래일은 별도 `/trade/start`(Phase 6
  스케줄러)로 새 `trade_run`을 연다 — idle로 계속 살아있지 않는다

### 6-5. 베이스 클래스 정지 계약 (선행 확장 — C2)

`BackgroundRunService`는 현재 정지 API가 없다(단발 배치 전제). 트레이딩은 장중
상시 루프 + 외부 킬 스위치가 필요하므로 **베이스에 협조적 정지 계약을 추가**한다
(TradingService에만 얹으면 CLAUDE.md 규칙 2 위반 — 땜빵):

- `request_stop(mode)` — `asyncio.Event`류 신호를 세운다. `mode ∈
  {STOP_NEW_ENTRIES, LIQUIDATE_ALL}`(단일 `_running` 불리언으로는 두 모드 표현
  불가 — 개발자 #1)
- `_run()`은 **폴링 사이클 경계에서** 정지 신호를 확인한다. **주문 발행 같은
  원자적 구간에서는 신호를 무시**하고 완료 후 다음 사이클에서 확인한다 —
  `place_order` 도중 `task.cancel()`로 끊기면 브로커엔 주문이 나갔는데 DB엔 기록이
  없는 고아 주문이 생긴다(아키텍처 #2)
- 기존 3서비스(collection/scoring/analysis)는 이 계약을 쓰지 않아도 동작 불변
  (정지 신호를 확인하지 않는 단발 `_run()`은 그대로)
- 이 확장은 §10의 "BackgroundRunService 승격"(started_at/finished_at·now 주입)과
  **함께** 처리한다

### 6-6. 재기동 reconciliation (`reconcile.py` — C1)

`_running`은 인스턴스 필드라 프로세스 재시작 시 무조건 False로 초기화된다. 스케줄러
없는 P5 시점에 운영자가 `/trade/start`를 다시 누르지 않으면 실보유 포지션이 손절
감시 밖에 방치된다 — 클라이언트측 TP/SL 전제가 무너진다.

**`_run()` 최초 진입부**(가드 통과 후. `_on_accepted()`가 아님 — 그 훅은 "예외
금지" 계약이라 실패 가능한 I/O 배치 불가, 아키텍처 #5)에서:

1. DB의 미종결 포지션(`PENDING_ENTRY`/`ENTERED`/`EXITING`/**`EXIT_FAILED`**) 조회 —
   EXIT_FAILED는 '실패로 고정됐지만 실보유가 잔존할 수 있는' 상태라 재기동 시
   반드시 재확인한다(P5-T5 아키텍트 — 저장소 open_positions와 정합)
2. `get_open_orders()` + `get_balance()`로 **브로커 실제 상태를 ground truth로** 대조.
   분기(전수 나열 — 유한하게 닫는다):
   - ① `PENDING_ENTRY`인데 브로커엔 이미 체결 완료 → `ENTERED`로 승격, 감시 편입
   - ② `PENDING_ENTRY`(`LIMIT_SUBMITTED`)인데 미체결 지정가 주문 생존 → 감시 재개
   - ③ **`PENDING_ENTRY`(`CANCEL_REQUESTED`)인데 브로커에 주문도 체결도 없음**
     (지정가 취소는 됐고 시장가 재발주 전 사망 — 고아 취소, v3 아키텍처 신규 #1)
     → **`ENTRY_FAILED`로 확정하고 알람.** 시장가 자동 재개는 하지 않는다(신호
     시점이 이미 지났을 수 있어 §6-3.0 갭 가드를 우회하는 신규 진입이 됨)
   - ④ `EXITING`인데 브로커엔 이미 청산됨 → `CLOSED` 확정
   - ⑤ **`EXITING`(`ExitPhase.LIMIT_SUBMITTED`)인데 익절 지정가 주문 생존**
     (v3 개발자 신규 #1) → 남은 `exit_limit_timeout_sec`를 무시하고 **즉시
     재평가**: 청산 사유가 여전히 유효하면 시장가로 마무리, 아니면 감시 복귀.
     청산은 진입과 달리 신호 시점 제약이 없으므로 재개해도 안전
   - ⑥ DB엔 없는데 브로커 잔고에 있음 → **경고**(수동 개입 필요, 상태 API 노출)
   - ⑦ **`EXIT_FAILED`인데 브로커 잔고에 보유 없음** → `CLOSED` 확정(직전
     매도가 실제로는 나갔던 것 — 확인 실패였을 뿐). **보유 있음** → 자동
     재청산하지 않고 **경고**(EXIT_FAILED는 이미 재시도 소진 상태 — 무한
     재시도 루프 금지, 수동 개입 대상. §6-7 warnings 노출)(P5-T5 아키텍트)
   - **시장가 미확정 케이스(v3.1 아키텍처·개발자 공통):** `PENDING_ENTRY
     (MARKET_SUBMITTED)` 또는 시장가 청산(`EXITING`, 손절·트레일링·기간초과 —
     `ExitPhase` 없음)에서 시장가 주문이 아직 미체결인 채 재기동된 경우는, 위
     ①/④(체결·청산 완료)와 ②(미체결 주문 생존)에 **브로커 실제 상태 기준으로
     자연 흡수**된다 — 시장가는 체결이 거의 즉시라 재기동 시점엔 대개 체결
     완료(①/④)이고, 드물게 미체결이면 ②의 "미체결 주문 생존 → 감시 재개"로
     처리. 별도 분기 불필요하나 "전수 나열"이 이 경로를 포함함을 명시
3. **진입 창 경계(v3 아키텍처 신규 #2):** reconcile은 **감시·청산만 재개하고
   신규 진입성 주문은 하지 않는다.** ②의 미체결 지정가는 진입 창(09:05~09:30)
   밖이면 **취소만** 하고 시장가 재발주로 넘어가지 않는다(취소는 신규 진입이
   아니지만 시장가 재발주는 실질적 신규 진입이므로 §6-3.7과 충돌). 재기동 후
   신규 진입은 오직 `/trade/start` + 진입 창 안에서만
4. 대조 판정 함수(`reconcile_decide` — **순수**, 입력: DB 상태 + 브로커 상태 →
   출력: 조정 액션 목록)와 그 액션을 적용하는 오케스트레이션 함수를 **파일 안에서
   이름으로 명확히 구분**한다(개발자 Minor — `reconcile.py`만 순수/부수효과 혼재).
   조회·쓰기는 주입 — 순수부는 테스트 가능하게

이 절차는 상태기계(§6-1)의 **시작 조건 자체**를 바꾸는 설계 결정이므로 스펙에 박는다.

## 7. 비용 모델 (`costs.py`) — SSOT 경계 + 시장별 비대칭 (트레이더 #5)

두 값을 **명시적으로 분리**(아키텍트 이월 해소):

| 용도 | 값 | 위치 |
|---|---|---|
| LLM 프롬프트 서술용 **근사** 왕복비용 | `AnalysisConfig.round_trip_cost_pct`(0.25) | `domain/analysis/config.py`(기존) |
| 실제 **체결비용 계산**(손익) | `TradingConfig`의 수수료·세율 필드(§6-2) | `domain/trading/costs.py`(신규) |

**`costs.py`가 반영해야 할 한국 시장 비대칭(트레이더 #5):**
- 위탁수수료는 **매수·매도 양쪽**에 부과 (`commission_buy_pct`/`commission_sell_pct`)
- 증권거래세+농특세는 **매도에만** 부과
- **코스피**(거래세 0.03%+농특세 0.15%)와 **코스닥**(거래세만) 세율이 다름 —
  `Instrument.market`(broker.py:21)으로 분기
- **ETF는 증권거래세 면제** — `market == "etf"` 분기
- 세율은 연도별 개정 → 값은 `TradingConfig` 설정값, §4에서 확정

**청산 판정 기준(트레이더 #5):** `exit_rules`의 `stop_loss_pct`/`take_profit_pct`는
**그로스 가격 기준**(비용 미반영)으로 정의한다. 비용은 **P&L 기록(§9)에만** 반영한다.
근거: 손절/익절 트리거를 넷 기준으로 하면 종목·시장마다 트리거 가격이 달라져
감시 로직이 복잡해지고, AI 필터(P4)는 이미 `round_trip_cost_pct`로 비용 엣지를
감안해 후보를 걸러냈다. **이 그로스/넷 경계를 §6-2에 명문화**해 구현자가 혼동하지
않게 한다.

  버그로 **첫 주문 한 건**이 상한을 넘는 경우를 못 막는다. 이 하드 캡이 그 구멍을
  닫는다
- **일일 주문 건수 상한**(`max_daily_orders`) — 초과 시 엔진 자동 정지 + 상태 노출
- **일일 주문 금액 상한**(`max_daily_order_krw`)
- **동일 종목 재진입 쿨다운**(`reentry_cooldown_min`) — 청산 직후 재매수 루프 차단
- **킬 스위치** `POST /trade/stop`(§8-1-b)
- **3자 상호 배제**(아키텍처 #3, 개발자 #9) — `conflict_check`는 pairwise 배선이라
  trading 추가 시 **양방향**으로 배선한다:
  - `main.py`에서 collection/scoring/trading이 서로를 참조하도록 3자 wiring
  - `api/collect.py`·`api/score.py`의 명시적 `is_running()` 가드에 trading 체크 추가
  - **양방향 확정:** 트레이딩 실행 중 collection/scoring 시작도 거부한다(수집의
    대량 TR 호출이 주문·손절을 지연시킬 수 있음 — §11-5). 편도 배선 금지

#### 8-1-b. 킬 스위치 시맨틱 (보안 #7)

- `mode` 파라미터: `STOP_NEW_ENTRIES`(신규 진입만 중단, 보유분 감시 유지) /
  `LIQUIDATE_ALL`(보유분까지 청산). **무조건 전량 청산을 기본으로 두지 않는다** —
  그 자체가 위험한 동작
- **`LIQUIDATE_ALL` 종료 조건(v3 개발자 신규 #2):** 청산 주문 발행 후에도 **전
  포지션이 `CLOSED` 또는 `EXIT_FAILED`로 종결될 때까지 폴링을 계속**하고, 그
  후에만 `_run()`이 반환한다. 발행 직후 루프를 끝내면 방금 낸 청산 주문(특히
  익절 지정가 대기)이 감시 밖에 놓여 §6-6 C1에서 고친 방치 문제가 재발한다.
  즉 `LIQUIDATE_ALL`은 "청산 지시 후 즉시 종료"가 아니라 "청산 완결까지 감시 지속"
- **마감시각 강제 종결(v3.1 아키텍처 신규):** 단 §6-4의 일별 종료(15:30 정상
  반환)와 충돌하지 않도록, **15:30 도달 시점에 `LIQUIDATE_ALL`이 미종결이면
  잔여 포지션을 `EXIT_FAILED`(알람)로 강제 확정하고 `_run()`을 반환한다.** 정규장
  마감 후에는 시장가 주문도 접수 불가(호가 없음)라 "완결까지 폴링"이 무한 대기로
  흐를 수 있기 때문. 이 규칙이 없으면 `_run()`이 늘어져 §8-1 3자 배타 배선상
  야간 배치(collection/scoring)까지 밀린다
- **응답 시맨틱:** `POST /trade/stop`은 정지 **요청 접수(202)**를 의미한다. 실제
  정지는 다음 폴링 사이클 경계에서 반영된다(§6-5 — 원자적 주문 구간은 선점 안 함).
  호출자가 "즉시 멈췄다"고 오판하지 않도록 상태 API에서 `status=stopping →
  stopped` 전이를 확인하게 한다
- **진입 대기창 선점:** 60초 지정가 미체결 대기 도중 킬 스위치가 오면 그 대기를
  중단하고 취소한다(다음 사이클 경계에서)
- **재시작 시 기본 상태:** 프로세스 재시작은 "정지" 상태로 시작한다(`/trade/start`
  전까지 자동 진입 없음). reconcile(§6-6)은 **감시만** 재개하고 신규 진입은 하지
  않는다

## 11. 리스크

1. **G1 불발 시 결정 #27 붕괴** — 단건 조회만 가능하면 감시 주기 = 종목 수 × 1초.
   최대 5종목(결정 #30)이면 5초 주기로 완화되나, 손절 지연 5초 수용 여부는 사용자
   재결정. 이 경우 `poll_interval_sec`의 의미도 재정의 필요(아키텍처 #9)
2. **모의 체결 시뮬레이션의 현실성 미지수** — 모의 체결은 실제 호가 잔량을 반영
   안 할 수 있다. 모의 체결률·슬리피지를 실전 근거로 삼지 않는다
3. **일봉 신호 + 장중 집행의 시점 괴리** — §6-3 0단계 갭 가드로 완화
4. **모의 피드 지연**(CLAUDE.md §5) — 상위 파이프라인이 게이트에서 막히면 진입
   신호가 안 나온다. 트레이딩 엔진 단독 테스트 경로를 확보
5. **레이트리밋 경합** — 주문·시세·잔고 버킷 분리 여부를 G2에서 확인. 같은
   버킷이면 감시 폴링이 손절 주문을 지연시킬 수 있어 **주문 우선순위** 설계 필요
   (결정 #14로 P5에 이관된 항목). §8-1 3자 상호배제가 수집발 경합을 차단
6. **동시호가·VI 구간 체결 지연**(트레이더 #6·#14) — §6-4 백오프로 대응하되,
   "즉시 체결" 가정이 깨지는 구간이 존재함을 수용
7. **배당락/액면분할 중 보유**(트레이더 #15, 비범위) — 보유 중 액면분할 시
   `avg_price`/`quantity`가 stale해져 오탐 청산 가능. 확률 낮음(오버나이트 위주),
   P5 비범위로 명문화
8. **결정 #29 v2 파생 리스크(v3, 트레이더 ①·②)** — 트레일링을 유일한 상방 관리
   수단으로 승격시킨 데 따른 두 부작용:
   - **휘핑쏘** — 활성화 경계의 노이즈성 되돌림 조기 청산. 결정 #35(2단계 폭)로
     완화하되, `trailing_activate_pct`/`trailing_stop_wide_pct`/`trailing_widen_
     until_pct`의 관계는 **백테스트 인프라 확보 후 캘리브레이션으로 기본값
     확정** 필요(현재는 설정값·논리적 기본치). P3 `simulation`은 스코어링용이라
     체결·비용 모델이 없어 그대로 못 씀 — 별도 이월
   - **무기한 보유** — 결정 #34(`max_holding_days`=10)로 상한을 둬 다중 갭 누적을
     차단. 다만 상한 내에서도 원 신호(P3/P4)의 유효성 재검증은 하지 않음 —
     보유 중 신호 재평가는 P5 비범위(향후 개선 후보)
