## 3. 데이터 기반 정비 (Phase 2 수집 수정 3건)

PRE-GATE 실측(2026-07-18, `.superpowers/sdd/p3-pregate-*.txt`)으로 확인된 결함의 해소.

### 3-1. 업종 소속 다대다 테이블

- 신규 테이블 `sector_memberships(sector_code FK, symbol FK, PK(sector_code, symbol))`.
- 수집 sectors 단계가 65개 그룹 **전부**의 ka20002 멤버를 그대로 저장 (덮어쓰기 없음).
  기존의 집계 업종 코드/이름 마커 필터와 50% 캐너리 로직은 **제거** — 필터링 책임이
  소비 시점(분류 기반)으로 이동하므로 수집은 단순 보존만 한다.
- `instruments.sector_code` 칼럼(+FK)은 **삭제** — last-write-wins로 손상된 데이터이며
  소비자가 아직 없다. 별도 데이터 복구 없이 수정 배포 후 `POST /collect` 1회로
  새 구조가 채워진다 (일봉 멱등).
- 멤버십 저장은 실행(run) 단위 전체 교체(delete-and-insert 또는 upsert+정리) —
  탈퇴/편출 반영을 위해 이전 실행의 소속이 남지 않아야 한다.

## 4. 스코어링 파이프라인

모듈 배치 (기존 계층 원칙 유지, 브로커 호출 없음 — DB만 읽음):

```
domain/scoring/
  config.py      # ScoringConfig dataclass (모든 파라미터 + 기본값)
  indicators.py  # 이동평균, 기간 수익률, N일 최고가, 평균 거래량 (순수 함수)
  strategies.py  # Strategy 프로토콜 + 모멘텀/눌림목/돌파 구현
  simulation.py  # 과거 신호 시뮬레이션 → 적합도(평균수익률·승률·횟수)
  engine.py      # ScoringEngine: 순수 계산 (입력 데이터 → 결과 객체)
  service.py     # ScoringService: 오케스트레이션 (게이트·저장·진행 상태)
store/scoring_store.py
api/score.py
```

### 4-1. 신선도 게이트

- 기준일 = `market_calendar.scoring_reference_date(now)` — **오늘 이전의 마지막
  평일**(수집 재개 판정에 쓰는 `previous_weekday`와 달리 오늘을 포함하지 않는다).
  스코어링은 자정 배치라 "어젯밤 수집분(전 거래일 종가)"이 최신인데
  `previous_weekday`(오늘이 평일이면 오늘)를 쓰면 매 자정 실행이 전 종목을
  낡음 판정해 100% 실패한다(T7 패널 트레이더 Critical). 당일 저녁(수집 직후)
  실행 시 하루 낡은 데이터가 게이트를 통과하는 트레이드오프가 있으나
  fail-safe 방향이며, Phase 6 스케줄러의 수집→스코어링 순서 강제로 소멸한다.
- 유니버스 종목별 최신 봉 날짜 < 기준일이면 그 종목은 **개별 제외**(집계에 기록).
- 제외 비율 > **5%** (설정값)이면 실행 전체를 `failed`로 마감, 사유 "stale data —
  run collection first". 소수의 개별 정체 종목(수집 실패 종목 등)이 전체를 영구
  차단하지 않으면서, 시스템적 낡음(수집 누락)은 확실히 차단한다.

### 4-2. 유니버스 필터

`market ∈ {kospi, kosdaq}` AND `is_active` AND `audit_info == "정상"` AND
`state`에 "거래정지"·"관리종목" 미포함. 실측 기준 약 2,500종목 통과 예상
(보통주 2,740 − 비정상 233).

### 4-3. 섹터 강도 (1단계)

- 대상: `group_type == "industry"`인 업종. 유니버스 통과 소속 종목이
  `min_sector_members`(기본 5) 미만인 업종은 제외.
- 업종 수익률 = 소속 종목(유니버스 통과분)의 기간 수익률 **동일가중 평균**.
- 섹터 점수 = R20 × 0.5 + R60 × 0.3 + R5 × 0.2 (Rn = n거래일 수익률).
- 전 업종 점수를 min-max 정규화([0,1], 실행 내) 후 상위 **K=5** 업종 진출.

> **주의:** 상대 정규화라 전 업종 하락장에서도 상위 K가 선정된다(시장
> 국면 필터는 Phase 5 설계 항목). 동일가중 평균은 소형 업종에서 단일
> 종목 급등에 민감(`min_sector_members=5` — Phase 4/5 파라미터 재검토
> 항목).

### 4-4. 3전략 신호 + 적합도 (2단계)

진출 업종의 유니버스 종목마다, 전략별로:

**(a) 오늘 신호** (기본값, 전부 설정화):

| 전략 | 신호 조건 |
|---|---|
| 모멘텀 | 종가 > MA20 > MA60 (정배열) AND R20 > 0 |
| 눌림목 | 종가 > MA60 AND MA20 상승(MA20[t] > MA20[t-5]) AND 직전 5일 내 최고 종가 > 당일 종가 (조정 존재) AND 당일 종가가 MA20 ±3% 밴드 내 |
| 돌파 | 종가 > 직전 60일 최고가(당일 제외) AND 당일 거래량 ≥ 20일 평균 거래량 × 1.5 |

**(b) 과거 6개월 적합도 시뮬레이션:**
- 과거 각 거래일 t에 대해 신호를 평가(지표 계산 가능 구간만), 신호 발생 시
  "t+1 시가 매수 → 매수일 포함 10거래일째 종가 청산" 수익률 기록.
- 잔여 봉 부족(신호가 너무 최근)인 발생 건은 표본에서 제외. 중복 보유 허용
  (발생 건별 독립 시뮬레이션 — 표본 수 확보 우선, 단순성).
- 적합도 = 평균수익률과 승률. **신호 발생 3회 미만이면 전략 점수 0** (표본 부족
  과대평가 방지).
- 전략 원점수 = 평균수익률 × 승률 → 실행 내 min-max 정규화([0,1]).

#### §4-4-b 한계 (T5 패널 트레이더 확인 — Phase 4/5에서 재확인 필수)

적합도(평균수익률·승률)는 다음 단순화 위의 값이다. Phase 4(AI 판단 재료)와
Phase 5(실거래 임계치) 소비 시 이 한계를 전제로 해석해야 한다:
1. **이상적 체결 가정** — 익일 시가 무조건 체결로 계산. 특히 돌파 전략은
   신호일 특성상 갭상승/상한가로 실제 체결가가 불리하거나 체결 불가일 수
   있어 적합도가 낙관적으로 편향될 수 있다.
2. **중복 보유 자기상관** — 추세 구간에서 연일 발생한 신호를 각각 독립
   표본으로 계상하므로, min_signal_occurrences=3을 충족해도 사실상 독립
   사건 1회일 수 있다.
3. **거래비용 미반영** — raw 가격 수익률(왕복 수수료+거래세 약 0.2~0.3%p
   미차감). 평균수익률이 이 비용 이하인 전략은 실질 손실 전략일 수 있다.
4. **상대 정규화와 게이트 상호작용** — 게이트(표본<3)된 전략 점수는 정규화
   후에도 0으로 클램프된다(구현 규칙 — 클램프 없이 순수 min-max만 적용하면
   표본 부족(원점수 0)이 검증된 음의 성과보다 유리해지는 역전이 발생하기
   때문). 점수는 실행 내 상대값이며 절대 신뢰도가 아니다 — Phase 4 AI는
   occurrences를 함께 참조할 것.

**(c) 종목 전략 점수** = 오늘 신호가 켜진 전략들의 정규화 적합도 합.
신호가 하나도 없는 종목은 후보 제외.

### 4-5. 합성·산출

- 최종 점수 = 섹터 점수(정규화) × 0.4 + 종목 전략 점수 × 0.6.
- 상위 **N=20** 종목을 그날의 후보로 저장. 후보마다 소속 업종·켜진 신호 목록·
  전략별 적합도(평균수익률·승률·발생 횟수) 동반 — Phase 4 AI의 판단 재료.

### 4-6. 설정값 일람 (ScoringConfig 기본값)

| 파라미터 | 기본값 | 파라미터 | 기본값 |
|---|---|---|---|
| 섹터 가중 (R20/R60/R5) | 0.5/0.3/0.2 | 진출 업종 수 K | 5 |
| 최종 가중 (섹터/종목) | 0.4/0.6 | 산출 후보 수 N | 20 |
| 보유 시뮬레이션 | 10거래일 | 최소 신호 표본 | 3회 |
| 업종 최소 구성 종목 | 5 | 신선도 제외 허용 비율 | 5% |
| MA 기간 | 20/60 | 돌파 박스 기간 | 60일 |

| 테이블 | 칼럼 (요지) |
|---|---|
| `score_runs` | id, started_at, finished_at, status(running/succeeded/failed), reference_date, universe_count, stale_excluded, failure_reason, config(JSON 스냅샷) |
| `score_sectors` | run_id FK, sector_code, sector_return_20/60/5, score, rank, selected(bool) |
| `scores` | run_id FK, symbol, rank, total_score, sector_code, sector_score, strategy_score(정규화 전 원값, 참고용), strategy_score_norm(후보 간 정규화값 — total_score 공식에 실제 사용된 값. `total_score = final_weight_sector×sector_score + final_weight_strategy×strategy_score_norm`이 저장값만으로 재현된다) |
| `score_details` | run_id FK, symbol, strategy(문자열), signal(bool), avg_return, win_rate, occurrences, score — 전략 추가 시 스키마 변경 없이 행만 증가 |

upsert 불필요 (run 단위 insert-only). sqlite/PostgreSQL 양쪽 지원 (기존 store 패턴).

## 6. API

| 엔드포인트 | 동작 |
|---|---|
| `POST /score` | 실행 시작 → 202 + run_id. 스코어링 실행 중이면 409. **수집 실행 중이어도 409** (소속·상태 갱신 도중의 반쪽 데이터 읽기 방지, 사유 메시지 포함) |
| `GET /score/status` | 진행 상황: run_id, status, stage(gate/sectors/strategies/finished), done/total, 실패 사유 |
| `GET /score/latest` | 최근 succeeded 실행의 상위 N 후보 + score_details + 진출 업종 — Phase 4/7 소비 인터페이스 |

서비스 시작은 CollectionService와 동일한 원자적 `start()` 패턴 (TOCTOU 없음,
태스크 강참조 유지, 종료 시 lifespan에서 취소).

## 7. 에러 처리

- 신선도 게이트 실패 → run `failed` + 사유 기록 (조용한 진행 금지).
- 종목 단위 계산 불가(봉 75개 미만 등) → 에러가 아닌 정상 제외 + 집계
  (수집의 종목별 실패 집계와 동일 패턴).
- 실행 중 예외/취소 → CollectionService와 동일한 예외 경계 (CancelledError/Exception
  → run을 failed로 마감, running 방치 금지).

## 8. 테스트 전략

- **단위(도메인):** indicators·전략 3종 신호·시뮬레이션을 손계산 기대값이 있는 합성
  봉 픽스처로 검증 (예: "이 봉 배열에서 눌림목 신호는 12일차에만 켜진다").
- **서비스:** 가짜 store로 게이트 거부(개별 제외/전체 실패), 유니버스 필터, 소규모
  end-to-end 결정적 순위.
- **store:** 신규 4테이블 + sector_memberships 왕복, sqlite/PostgreSQL 패리티.
- **수집 회귀:** 멤버십 전체 교체, state/audit_info 저장, 기존 수집 테스트 무결.
- **수용 기준:** §3-2 industry 중복 소속률 검증 + 실데이터 1회 실행(재수집 →
  스코어링)으로 상위 N 후보 산출 확인.
- 구현은 태스크별 4-에이전트 패널 리뷰(규칙 8), 서브에이전트 주도 개발로 진행.

## 9. 이후 페이즈 연계

- **Phase 4 (AI):** `GET /score/latest`(또는 store 직접)로 후보+근거 소비.
  **착수 전 선행 리팩터:** CollectionService/ScoringService의 공통 오케스트레이션
  스캐폴딩(원자적 start·conflict_check·예외 경계·진행 상태)을 공통 베이스로 추출
  — 두 번째 중복까지는 의도적 이연(T8 수용 검증 직전 동시 재구조화 회귀 리스크
  회피, T7 아키텍트 패널 조건부 수용), 세 번째 동형 서비스가 생기는 Phase 4가
  추출 적기.
- **Phase 5 (트레이딩):** TP/SL 정책 확정 후 시뮬레이션을 TP/SL 반영형으로 고도화
  (백로그). 유동성 하한 필터 재평가.
- **Phase 6 (스케줄러):** 수집 완료 → `POST /score` 순서 보장, 거래일 캘린더 도입
  시 신선도 게이트의 평일 근사 한계 소멸.

## 10. 리스크 — 무인증 쓰기 경로

`POST /score`·`POST /collect`는 무인증 쓰기 엔드포인트다 — localhost(127.0.0.1)
바인딩을 전제로 한 설계(P2 spec §1·§8과 동일 전제). 그러나 현재 CORS는
`allow_origins=["*"]`이며, 브라우저의 단순 cross-origin POST(본 요청이 preflight
없이 서버에 도달하는 "simple request" 조건을 충족)는 사전검사(OPTIONS) 없이
그대로 도달한다 — 악성 웹페이지가 사용자의 브라우저를 통해 이 엔드포인트를
드라이브-바이로 트리거할 수 있다는 뜻이다. Phase 5(실전 전환)·Phase 7(UI 노출)
착수 전 반드시 인증/오리진 제한 도입 여부를 재평가해야 한다(T7 패널 보안).
